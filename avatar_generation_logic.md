# Логика работы системы генерации аватаров

Этот документ описывает полный жизненный цикл данных: от нажатия кнопки «Создать проект» до получения готовых карточек сегментов целевой аудитории.

## Шаг 1. Сбор контекста (Формирование Брифа)
Все начинается на странице `new/page.tsx`.
1. **Ручной ввод**: Пользователь заполняет поля (Название продукта, Категория, Описание, Возраст, Доход, ГЕО и т.д.).
2. **Внешние источники (Опционально)**:
   - *Google Doc*: Пользователь может дать ссылку на бриф.
   - *Лендинг*: Если указан URL сайта, система делает запрос `/api/parse-landing`, который "выкачивает" текстовое содержимое страницы.
3. **Объединение**: Весь собранный текст склеивается в единый объект `briefData`. Этот объект станет главным контекстом (базой знаний) для ИИ на всех последующих этапах.

## Шаг 2. Быстрая сегментация (IDENTIFY_SEGMENTS)
Если пользователь не вписал свои названия сегментов вручную, система должна их придумать.
1. Отправляется запрос к ИИ с промптом `IDENTIFY_SEGMENTS`.
2. **Логика промпта**: 
   - ИИ получает `briefData`.
   - Задача: выдать 10 РАЗНЫХ вариантов целевой аудитории.
   - Жесткие ограничения стилистики: запрет на пафос, драму («Жертвы», «Заложницы»). Названия должны быть в стиле «Диагноза» или заголовка на форуме (например, «Мамы в декрете, которые хотят свой доход»).
3. **Ответ**: ИИ возвращает строгий JSON-массив из 10 элементов (название + краткая суть).
4. **Взаимодействие**: Пользователь видит 10 карточек на экране и выбирает от 1 до 3 самых релевантных.

## Шаг 3. Глубокое исследование (RESEARCH_SEGMENT)
Это самый тяжелый и важный этап. Происходит генерация психологических портретов для выбранных сегментов.
1. **Каскадный запуск**: Чтобы API Google не заблокировал нас за спам, запросы стартуют параллельно, но с задержкой:
   - 1-й аватар: старт сразу (0 сек)
   - 2-й аватар: старт через 3 секунды
   - 3-й аватар: старт через 6 секунд
2. **Логика промпта**:
   - Используется промпт `RESEARCH_SEGMENT`.
   - ИИ получает: весь `briefData` + `segmentName` (название конкретного сегмента, которое мы сейчас исследуем).
   - Задача: заполнить 10 глубоких блоков (Портрет, JTBD, Результаты, Боли, Страхи, Симптомы, Маркеры поведения, Мотивации, Возражения, CJM).
   - Требования: использовать «елочки» для цитат, писать живым языком форумов, приводить конкретные примеры.
3. **Контроль формата (Structured Outputs)**: Бэкенд передает в Google Gemini жесткую схему (JSON Schema). ИИ не может вернуть просто текст, он обязан распихать слова по нужным ячейкам JSON. Это гарантирует, что система не сломается при чтении ответа.
4. **Обработка ошибок**: Если ИИ дает сбой (например, не хватает токенов), система делает до 3-х попыток перезапросить конкретно этот аватар.

## Шаг 4. Сохранение и Дашборд
1. Успешно сгенерированные аватары собираются в массив.
2. Фронтенд отправляет запрос к `/api/projects` для сохранения всего объекта (Бриф + Аватары) в базу данных Supabase.
3. Пользователя перекидывает на страницу `[id]/page.tsx` (Дашборд проекта).

## Доступные возможности внутри Дашборда
После генерации у пользователя есть инструменты управления:
1. **Ручное редактирование**: Можно переименовать сегмент или изменить его краткое описание прямо в UI.
2. **Замена сегмента (Регенерация)**: 
   - Кнопка "Заменить сегмент через ИИ".
   - Система отправляет запрос на бэкенд с флагом `regenerateOne`.
   - ИИ смотрит на текущие названия аватаров и придумывает **совершенно новое** название, отличное от существующих.
   - Сразу после этого ИИ проводит для этого нового названия глубокое исследование (`RESEARCH_SEGMENT`) и заменяет старую карточку на новую.
3. **Генерация креативов**: Переход к следующему этапу (создание рекламных связок на основе готовых болей и страхов аватара).
